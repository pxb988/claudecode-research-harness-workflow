## 0. 这份指南给谁 · 你需要先准备什么

**适合你，如果：** 你已经会打开 Claude Code 并和它对话，研究方向是经济学 / 社会学 / 教育学 / 政治学 / 公共卫生等社科领域，但你**没用过**这套「研究流水线」，看到一堆 `/research-harness-*` 命令不知从何下手。

**读完你能做什么：** 独立把一个实证研究项目从「拿到原始数据」走到「交出一个别人能复现的复制包」，中间的清洗、合并、分析、审阅每一步都有脚本、有日志、有报告。

**开始前，确认三件事都就绪：**

1. **你会和 Claude Code 对话。** 能打开它、能输入命令、能看懂它的回复就够了。本指南不教 Claude Code 本身怎么装。
2. **你电脑上装了一种分析语言。** R、Stata、Python **任选其一**即可：
   - R：<https://cran.r-project.org/>
   - Stata：<https://www.stata.com/>
   - Python：<https://www.python.org/>

   主流程没指定语言时**默认用 R**；只有「论文复现模式」（第 4 章）需要 Python。这里不展开安装步骤——按官方说明装好、能在命令行调用即可。
3. **你已经在你的研究项目里装好了本插件。** 安装方法见 [README 的「快速上手」](../README.md#快速上手)，本指南不重复那段命令，避免两处对不上。

> 三项都齐了？很好，往下读。

## 1. 先建立直觉：这套工作流在帮你做什么

AI 现在能帮你写代码、清数据、跑回归、生成表格。问题是：**没有约束的 AI 做研究，很容易出看不见的错。** 比如——

- 同一个分析跑了两遍，样本数对不上；
- 正文里的系数是手抄的，后来和表格对不上；
- 因果话说得比识别策略能支撑的更满；
- 清洗时悄悄改了原始数据文件；
- merge（合并）其实失败了，却被总结成「分析完成」；
- 最后脚本跑不出论文里的数字，日志也找不到，数字来源成了谜。

这套工作流（我们叫它 **harness**，可以理解成「给 AI 套的安全带 + 自动质检」）干的事，就是**把上面每一个风险都变成一个明确的检查点**：原始数据被保护起来不许改、每次清洗都留痕、每次合并都出诊断报告、每个分析任务都得有脚本和日志、每句因果话都要标注强度、最后发布前还要独立审一遍。

它的核心信条只有一句：

> **No script, no log, no claim. —— 没有脚本、没有日志，就没有结论。**

整个流程分七个阶段，从头到尾是这样一条线：

```
setup    →  audit   →  clean   →  plan    →  work    →  review  →  release
定义问题    审查数据    清洗合并    制定计划    执行分析    审阅结果    发布复制包
```

- **setup** 把研究问题、数据、设计写成「研究合同」，并立好规矩（含保护原始数据）。
- **audit** 只看不动地审查原始数据，判断你的研究设计在这份数据上到底可不可行。
- **clean** 清洗、重编码、合并多个数据文件，每一步留痕、每次合并出诊断。
- **plan** 把研究想法拆成一条条可执行的分析任务，等你批准。
- **work** 真正跑分析：写脚本、跑出来、存日志和结果表图。
- **review** 独立审一遍：数字能不能追溯、因果话有没有说过头。
- **release** 审过之后，打包成别人能照着复现的复制包。

值得这点麻烦吗？值得。它换来的是：**你交出去的每一个数字，都能顺着脚本和日志查到出处。** 这正是「AI 助手」和「可信研究」的分水岭。

## 2. 几个你必须先认识的词

不用全背，混个脸熟，遇到时回来查即可。

- **项目目录** —— 你这个研究项目的文件夹。**你要在这个文件夹里启动 Claude Code**，而不是在插件仓库里。所有命令都对当前项目目录生效。
- **canonical 布局** —— 工作流约定的标准文件夹结构，一共 **8 个目录**，各装各的：

  | 目录 | 装什么 |
  |---|---|
  | `0.dofiles/` | 你的分析脚本（R / Stata / Python） |
  | `0.dofiles/logs/` | 每次脚本运行的日志 |
  | `1.rawdata/` | **原始数据（只读，永不写入）** |
  | `2.workdata/` | 中间清洗数据，可随时重新生成 |
  | `3.outdata/data/` | 分析就绪数据 + codebook |
  | `3.outdata/figures/` | 图 |
  | `3.outdata/tables/` | 表 |
  | `4.reports/` | 审查 / 清洗 / 合并 / 审阅 / 复现各种报告 |

- **hook（自动质检员）** —— 三个在后台自动运行的「守门员」，**全程生效，不管你在哪一步**：
  - 你想往 `1.rawdata/` 写东西 → 被**拦下**（原始数据只读）。
  - 你想把数据文件 `git add` 进版本库 → 被**拦下**（数据不进库）。
  - 你跑了脚本却没留日志 → 它**提醒**你去补日志。
- **agent（替你干活的两个角色）** —— `analyst` 在 work 阶段替你写脚本、跑分析、存日志，**从不编数字、从不改原始数据**；`reviewer` 在 review 阶段**只读审阅**，从不跑代码、从不改文件。
- **`cc:` 状态标记** —— 每个分析任务的进度标签：`cc:todo`（没开始）、`cc:wip`（进行中）、`cc:done`（完成且证据齐全）、`cc:blocked`（卡住，原因已记）、`cc:infeasible`（数据/设计不支持，已说明）。
- **证据链** —— 一句话：你交出的每个数字，都能顺着「脚本 → 日志 → 输出文件」一路查回去。这是整套流程的灵魂。

## 3. 手把手跑完一个完整项目（主线）

这一章我们用仓库**自带的现成例子** `examples/econometrics-replication` 完整跑一遍七阶段。它是一份**合成的**双重差分（DiD）数据——研究「某地区职业培训项目是否提高了家庭年收入」。**数据是随机生成的，数字没有实际意义，纯粹供你练手。**

### 3.0 准备工作（很重要，别跳）

**第一步：把例子复制到一个干净的工作目录。** 不要直接在插件仓库里跑——那会污染仓库，而且原始数据保护规则会和示例夹具打架。

```bash
# Mac / Linux（在插件仓库目录下执行）
cp -r examples/econometrics-replication ~/my-first-study
cd ~/my-first-study
```

```text
# Windows（PowerShell）
Copy-Item -Recurse examples\econometrics-replication $HOME\my-first-study
cd $HOME\my-first-study
```

**第二步：在 `my-first-study` 这个目录里启动 Claude Code。** 之后所有 `/research-harness-*` 命令都对这个目录生效。

> **小提示：** 这个例子已经**预填好了** `study_spec.md`（研究合同，DiD 设计现成的）。所以待会儿的 `setup` 这步，主要是帮你建好文件夹结构、装好治理规则、给原始数据上只读锁，而**不是**从零让你写研究合同。换成你自己的真实项目时，`setup` 才会带你从头写 spec。

下面七个阶段，每一节我都用同一个模板讲：**① 你输入什么 → ② 会发生什么 → ③ 跑完该出现哪些文件 → ④ 你该检查什么 → ⑤ 这步在保护你什么。**

---

### 阶段 1 · setup（定义问题、立规矩）

**① 你输入**
```text
/research-harness-setup
```

**② 会发生什么**
Claude 建立 canonical 布局（上面那 8 个目录）、交付项目治理 `CLAUDE.md`、写好保护数据的 `.gitignore` 与权限设置；并在原始数据放进 `1.rawdata/` 后，给它们上一道操作系统级的**只读锁**。本例 `study_spec.md` 已预填，setup 会沿用它。

**③ 跑完该出现哪些文件**
8 个目录就位；`1.rawdata/` 里的 `panel_outcomes.csv`、`policy_timing.csv` 变成只读；项目根多出治理 `CLAUDE.md`。

**④ 你该检查什么**
确认目录结构出现了、`study_spec.md` 在、原始数据还在 `1.rawdata/`。看一眼 `study_spec.md` 里的研究问题和设计（DiD），心里有数。

**⑤ 这步在保护你什么**
从一开始就立好「研究合同」和「原始数据只读」两条底线——后面所有阶段都靠它们兜底。

---

### 阶段 2 · audit（只看不动地审数据）

**① 你输入**
```text
/research-harness-audit
```

**② 会发生什么**
Claude **只读**地检查 `1.rawdata/` 里的两份文件：有哪些变量、缺失情况、能不能对上你的 DiD 设计、两份文件靠什么键合并。**它绝不修改原始数据。**

**③ 跑完该出现哪些文件**
`4.reports/data_audit_report.md`。

**④ 你该检查什么**
打开 `data_audit_report.md`，看它怎么评估「这份数据能不能支撑 DiD」、它认为两份文件该用什么键合并。如果它说某处不清楚而停下来问你——**回答它，别让它瞎猜**。

**⑤ 这步在保护你什么**
在你动手清洗之前，先确认「数据真的撑得起你的研究设计」，避免白忙一场。

---

### 阶段 3 · clean（清洗 + 合并，重点看 merge 报告）

**① 你输入**
```text
/research-harness-clean
```

**② 会发生什么**
Claude 清洗两份数据并把它们**合并**成一个分析面板：`panel_outcomes.csv`（每户每年一行）和 `policy_timing.csv`（每个地区每年一行）按「地区 × 年份」合并。这是一个 `m:1`（多对一）合并。合并后还会派生两个变量：`treated = (policy_active == 1)`、`post = (year >= treat_year)`。**每一步 drop / recode / reshape / merge 都留痕。**

**③ 跑完该出现哪些文件**
`2.workdata/`（中间数据）、`3.outdata/data/panel_analysis.csv`（合并后的分析面板）、`4.reports/data_cleaning_report.md`（清洗报告）、`4.reports/merge_report.md`（合并诊断）。

**④ 你该检查什么 —— 重点学会读 merge 报告**
打开 `merge_report.md`，本例你应该看到类似：

> **600 行面板 × 20 行政策 → 合并后 600 行，匹配率 100%。**

这就是一次健康的 `m:1` 合并：面板的每一行都在政策表里找到了对应的「地区 × 年份」。merge 报告必报这几样，逐个看：**合并键 / 合并前后行数 / 匹配与未匹配数 / 重复键诊断 / 变量冲突 / 未解决问题**。如果匹配率不是 100%、或出现未匹配行，别忽略——那是真问题。

**⑤ 这步在保护你什么**
合并是实证研究最容易出隐性错误的地方。强制出诊断报告，让「合错了」无所遁形。合并键不清楚时，工作流会停下来问你，**不靠猜测继续**。

---

### 阶段 4 · plan（把想法拆成可执行任务）

**① 你输入**
```text
/research-harness-plan
```

**② 会发生什么**
Claude 根据 `study_spec.md` + 审查报告 + 清洗后的数据结构，生成 `analysis_plan.md`——一张**任务台账**，把研究拆成一条条带编号的分析任务（如 1.1 描述统计、2.1 主回归、3.1 稳健性）。

**③ 跑完该出现哪些文件**
`analysis_plan.md`（项目根）。

**④ 你该检查什么**
通读 `analysis_plan.md` 的任务列表，确认它们对应你想要的结果（描述统计、pre-trend 检验、主 DiD、稳健性）。**这是一道人工审批门：你批准后，work 阶段才该执行。**

**⑤ 这步在保护你什么**
先把「要做什么」和你对齐，再动手跑——避免 AI 跑了一堆你其实不想要的东西。

---

### 阶段 5 · work（真正跑分析）

本例有三个任务要跑，逐个来：

**① 你输入**（一个一个跑）
```text
/research-harness-work 1.1
/research-harness-work 2.1
/research-harness-work 3.1
```
（`1.1` 描述统计与 pre-trend、`2.1` 主 DiD 回归、`3.1` 稳健性。也可以 `/research-harness-work all` 一次跑完所有已批准任务。）

**② 会发生什么**
`analyst` agent 为该任务写脚本、运行它、把日志存进 `0.dofiles/logs/`、把结果表图存进 `3.outdata/`，并把证据路径回填进 `analysis_plan.md`。**它从不编数字。**

**③ 跑完该出现哪些文件**
`0.dofiles/logs/` 里多出该次运行的日志；`3.outdata/tables/`、`3.outdata/figures/` 里多出对应表图。

**④ 你该检查什么 —— 记住 `cc:done` 的五个条件**
一个任务能被标成 `cc:done`，**必须同时满足五条**，缺一不可：

1. 脚本存在；
2. 脚本已经真的运行过；
3. 日志存在；
4. 预期的输出文件（表/图）存在；
5. 证据路径已写进 `analysis_plan.md`。

如果某任务反复失败，它会被标成 `cc:blocked` 并记录原因——**不会被假装跑完**。

**⑤ 这步在保护你什么**
`remind-evidence` 这个 hook 会盯着：你跑了脚本却没留日志，它就提醒你。五条件 + 日志强制，保证每个数字都查得到来源。

---

### 阶段 6 · review（独立审一遍）

**① 你输入**
```text
/research-harness-review
```

**② 会发生什么**
`reviewer` agent **只读**地审你的全部产出：识别策略对不对、每个数字能不能追溯到日志、因果话有没有说过头、清洗与合并完不完整、原始数据有没有被保护。**它从不跑代码、从不改文件。**

**③ 跑完该出现哪些文件**
`4.reports/review_report.md`。

**④ 你该检查什么 —— 看它给出的三选一结论**
review 只会得出三种结论之一：

- **`APPROVE`** —— 证据链足够，可以发布。
- **`REQUEST_CHANGES`** —— 还有重要问题，改完再审。
- **`BLOCK`** —— 严重失败（比如数字无法验证、因果声明没有支撑）。

本例里，reviewer 还会专门检查：脚本头是否带 `[quasi-experimental: DiD]` 标签、`analysis_plan.md` 里是否有 pre-trend 检验任务、结果表里的每个系数是否都能在日志文件里找到。**任何没带 DiD 标签却声称「因果效应」的地方，都会被标为重要问题。**

**⑤ 这步在保护你什么**
在你对外发布前，有一道独立、只读、较真的审查，专挑「数字对不上、话说过头」。

---

### 阶段 7 · release（打包复制包）

**① 你输入**
```text
/research-harness-release
```

**② 会发生什么**
**只有 review = `APPROVE` 时**才允许发布。Claude 把脚本、就绪数据、表图、报告组装成一个别人能照着复现的复制包，并生成复现说明。未经批准的机密原始数据**不会**进包。

**③ 跑完该出现哪些文件**
项目根出现 `release/`（复制包）+ `4.reports/reproducibility_report.md`（复现说明与证据索引）。注意 `release/` 不是 setup 建的，是这一步按需创建的。

**④ 你该检查什么**
打开 `reproducibility_report.md`，确认它把「用什么脚本、按什么顺序、能跑出哪些结果」说清楚了；确认包里没有不该外传的原始数据。

**⑤ 这步在保护你什么**
「未审过不许发」+「机密数据不入包」，让你交出去的东西既可复现、又不泄密。

---

🎉 走到这里，你已经完整体验了一个研究项目从数据到复制包的全过程。**真正的功力在于「每一步你都知道该检查什么」**——这正是上面每节第 ④ 段在训练你的。

## 4. 论文复现模式（replicate）

这是**七阶段主流程之外的一个独立模式**，不需要先走完整的 clean。

**它干什么：** 给定一篇已发表的论文，直接读原始调查面板的元数据，提取出与论文「样本筛选、变量构造、编码」一致的子样本。

**什么时候用：**
- 复现某篇论文的基准回归前，先还原它的样本和变量定义；
- 用同一个原始数据库，做不同样本 / 变量的比较或扩展；
- 验证论文里的描述统计能不能从原始数据复现出来。

**标准流程是三个脚本：**

```text
*_10_discover_vars.py      # 读原始 .dta 元数据，定位论文所需变量的真实名称
*_11_<paper>_subsample.py  # 按论文的筛选条件和变量定义，提取子样本
*_12_<paper>_codebook.py   # 生成 codebook，含论文变量定义和均值对比
```

所有输出写进 `3.outdata/data/`：变量元数据（`*_var_list.csv`）、分析就绪子样本（`*_subsample.parquet` / `.csv`）、对比 codebook（`*_codebook.csv`）。

**怎么判断复现得准不准：** codebook 会给每个构造出来的变量，标一个它和论文描述统计表的均值差距等级：

- **`CLOSE`** —— 和论文很接近，放心。
- **`MODERATE`** —— 有点差距，留意。
- **`DIFFERS`** —— 差得明显，**必须在 `notes` 列写清楚差距成因**，才能算就绪。

> 复现模式需要 Python（含读取 `.dta` 的依赖）。它适合多波次调查面板、含多个模块的大型微观数据、变量名跨波次有变化的数据集。

## 5. 模块化使用：不走完整七阶段时

你**不一定每次都从 setup 走到 release**。可以只用其中一段：

| 你想做什么 | 用哪个命令 |
|---|---|
| 只检查一个新数据集、不修改它 | `/research-harness-audit` |
| 清洗一个混乱的调查数据文件 | `/research-harness-audit` → `/research-harness-clean` |
| 合并个人 / 家庭 / 社区 / 政策时间文件 | `/research-harness-clean` |
| 按论文定义提取子样本和变量集 | 见第 4 章「论文复现模式」 |
| 把研究想法变成可执行实证计划 | `/research-harness-setup` → `/research-harness-plan` |
| 只运行已批准的某个回归 / 表格 / 图形任务 | `/research-harness-work <task-id>` |
| 检查已有结果是否可信 | `/research-harness-review` |
| 投稿前整理复制包 | `/research-harness-release` |

**跳阶段是有代价的。** 从中间进入时，每个命令都会提示你：跳过前面的阶段会损失哪些检查、增加哪些风险。例如 `clean` 可以单独用，但更稳的最小路径是 `audit → clean`——**如果 ID、缺失值编码、日期、单位或合并键还没搞清楚，别盲目清洗数据。**

> 不确定此刻该用哪个命令？在 Claude Code 里运行 `/workflow-guide`，它会替你导航。

## 6. 可累积知识层（references）

这套工作流自带一个**会越用越聪明**的知识库，技能在审查 / 清洗 / 复现时会**自动查阅**它：

- `docs/references/datasets/<id>.md` —— 某个数据集的文件命名、变量编码「坑」（当前已收录 CHARLS）。
- `docs/references/replications/<paper>.md` —— 某篇论文的「概念 → 变量」映射和公开目标均值（当前还只是一个可扩展的模板）。
- `docs/survey-pipeline-engineering.md` —— 宽面板的内存 / 类型处理经验。

**怎么把你的经验沉淀进去（清洗一个新数据库时最有用）：**

假设你清洗了一个新数据库、踩了一些命名和编码的坑。把这些经验固化成可复用知识，只要两步、**不用改任何技能代码**：

1. 新建一个文件 `docs/references/datasets/<新id>.md`，把这个数据库的文件命名规律、变量编码 gotchas 写进去。
2. 在 `docs/references/index.json` 里加**一条登记**。它的格式就是这样（照着 CHARLS 那条仿写）：

   ```json
   "datasets": {
     "charls": {
       "file": "datasets/charls.md",
       "title": "China Health and Retirement Longitudinal Study"
     },
     "你的新id": {
       "file": "datasets/你的新id.md",
       "title": "数据库全名"
     }
   }
   ```

登记完，**下次再清洗这个数据库，技能就会自动查到你写的经验**——你踩过的坑，以后（你或你的合作者）不用再踩。

## 7. 常见问题 · 出错怎么办

每条都按「**现象 → 为什么 → 怎么办**」给你。

**我想往 `1.rawdata/` 里写文件，被拦下来了。**
为什么：原始数据是只读的，`guard-raw-data` 这个 hook 在保护它。
怎么办：别写原始数据目录。把清洗后的结果写到 `2.workdata/`（中间数据）或 `3.outdata/`（成品）。

**我 `git add` 数据文件时被拒绝了。**
为什么：数据文件不进版本库，`guard-git-add` 在拦。
怎么办：只提交脚本和报告，数据留在本地。（`3.outdata/tables/` 和 `figures/` 里的成品表图是例外，允许提交。）

**Claude 把某个任务标成 `cc:infeasible`，或者说「这个做不了」。**
为什么：这是**设计，不是 bug**。数据或研究设计撑不起这个任务时，正确做法就是停下并说明，而不是硬编一个看起来合理的结果。
怎么办：看它记录的原因，据此调整研究设计、放宽/收紧样本，或换更合适的数据。

**我输入 `/research-harness-...` 命令，没反应 / 命令根本不出现。**
为什么：插件可能没在当前这个项目里启用。
怎么办：回 [README 的「快速上手」](../README.md#快速上手)确认安装和启用步骤；确认你是在**研究项目目录**里启动的 Claude Code，而不是别处。

**原始数据被操作系统只读锁住了，我想换一批数据怎么办？**
为什么：只读锁是为了防止原始数据被悄悄改动。
怎么办：**不要手动去改被锁的文件**。正确做法是把新的原始数据文件放进 `1.rawdata/`，再重新走 `setup` / `audit` 让流程重新锁定与审查。

**「没有日志就没有结论」这类提醒反复出现。**
为什么：`remind-evidence` hook 发现你跑了脚本却没留日志。
怎么办：让脚本运行时把输出留到 `0.dofiles/logs/`（例如把运行日志重定向到该目录的一个文件）。标任务完成前，先确认日志确实生成了。

**我中途想换分析语言（R ↔ Stata ↔ Python）。**
为什么：主流程的分析语言是可以换的（默认 R）。
怎么办：在 `study_spec.md` / `analysis_plan.md` 里把语言改清楚并保持前后一致，后续脚本就按新语言写。

## 8. 下一步：换成你自己的数据

练完例子，把它换成你真正的研究：

1. 把你的原始数据放进 `1.rawdata/`（替换掉示例数据）。
2. 改 `study_spec.md`：写清你的研究问题、数据、识别策略、变量和样本。
3. 从 `/research-harness-setup` 或 `/research-harness-audit` 开始，重走一遍七阶段。

想更深入了解每个阶段的目的与「交接契约」，读这几份：

- README 的「七阶段参考」表 —— 速查每阶段输入/产出/强制项。
- [`docs/RESEARCH-WORKFLOW.md`](RESEARCH-WORKFLOW.md) —— 各阶段目的与交接契约的完整说明（英文）。
- [`docs/INTEGRITY-RULES.md`](INTEGRITY-RULES.md) —— 研究诚信规则全文（保护原始数据、不编证据、标注因果声明等）。

记住那句话——**没有脚本、没有日志，就没有结论。** 跑得慢一点没关系，跑得能被追溯，才是真本事。
